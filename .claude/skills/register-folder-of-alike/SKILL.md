# Register a folder of alike entities

This skill exists because asking the user 847 sets of questions for 847 alike files makes them quit. Capture **one template** that the emitted build script expands into N `Dataset.model_validate({...})` calls in a `for` loop.

## When to invoke this (not register-dataset)

- User says "the raw images", "all the FASTQs", "the directory of plate scans" — anything plural.
- A folder contains many files of the same extension serving the same role.
- You see ≥ 5 alike files and the user has not given them individual descriptions.

If the count is small (1–4) and the user wants per-file descriptions, use `register-dataset` per file instead.

## Inputs you need (ask once, not per-file)

1. **Glob.** What files? Express as a path glob relative to project root: `raw/images/*.tif`, `data/fastq/**/*.fq.gz`. Show the user the expansion ("this matches 847 files — confirm?") and let them tighten the pattern.
2. **What kind of files** (description). One sentence. The build script will optionally interpolate `{basename}` / `{stem}` per file.
3. **Author / source.** Default crate-level author.
4. **Format.** Infer from the dominant extension; confirm with user.
5. **Raw input or generated by a step?** → `is_raw_input`.
6. **Per-item overrides?** Default: none. If the user says "all images except control_01.tif have label X", record `overrides: {"raw/images/control_01.tif": {"description": "..."}}`.

## GUID strategy — deterministic per-file

Bulk-group GUIDs are derived deterministically so they're stable across re-runs and across script edits:
```
ark:59853/<guid_prefix>-<slug(basename)>
```
No timestamp/random squid. The `guid_prefix` itself should be slug-safe (e.g. `dataset-microscopy-image`).

## Snapshot the file list

At registration time, expand the glob and store the resulting filenames in `snapshot_files`. This is what `plausibility-check` and `create-computation` reason about. The emitted build script re-evaluates the glob at run time (so files added later are picked up automatically), but the snapshot is what the wizard used to build the graph.

## Write to state

Append to `state.bulk_groups`:
```json
{
  "guid_prefix": "dataset-microscopy-image",
  "glob": "raw/images/*.tif",
  "template": {
    "name_template": "Microscopy image {basename}",
    "description_template": "Raw microscopy image {basename} from acquisition session",
    "author": "Lab Tech",
    "keywords": ["microscopy", "raw"],
    "format": "image/tiff",
    "version": "1.0",
    "datePublished": "2026-05-04"
  },
  "is_raw_input": true,
  "user_label": "raw microscopy images",
  "overrides": {},
  "snapshot_files": ["raw/images/control_01.tif", "raw/images/sample_001.tif", "..."]
}
```

Tell the user **one line**: `added 847 microscopy images as raw inputs (one template — expands at build time)`. Append `history`.

## Referring to a bulk group from a computation

Computations link to a bulk group by its `guid_prefix`, not by enumerating every per-file GUID. The build script handles expansion. See `create-computation`.
